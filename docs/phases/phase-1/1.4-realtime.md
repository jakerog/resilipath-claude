# Sub-Phase 1.4 — Real-Time Engine
**Version:** 0.2.0  
**Date:** 2026-03-29  

**Sub-Phase Title:** Real-Time Engine  
**Primary Concern:** Implement the live exercise update system using Firebase Realtime Database, and build the DR Bridge — the full-screen live summary view for exercise calls. No standard CRUD. No UI components beyond the Bridge. Real-time data flow only.  
**Dependencies:** 1.1 (schema), 1.2 (auth)  

---

## Architecture Decision: Firebase Realtime Database (not Firestore listeners)

For live exercise updates we use **Firebase Realtime Database** (not Firestore's `onSnapshot`), because:
- Realtime DB is optimized for high-frequency, low-latency write/read (sub-100ms)
- It uses a persistent WebSocket connection managed by the Firebase SDK
- Free tier is generous (1GB storage, 10GB/month transfer) and sufficient for MVP
- Firestore `onSnapshot` incurs a read charge per document per listener — at 200 concurrent participants watching 100 tasks, this would rapidly exhaust the free read quota

**Architecture:**
```
API writes task mutation → Firestore (source of truth)
                         → Firebase Realtime DB (live mirror for exercise)

Client reads from → Firestore (initial load)
                  → Realtime DB (live updates via Firebase SDK listener)
```

The Realtime DB holds a lightweight mirror of the active exercise state — only the fields needed for the live view:

```
/exercises/{exerciseId}/
  meta: { name, status, currentPhase, startedAt }
  stages/{stageId}/
    name, status, completedCount, totalCount
  tasks/{taskId}/
    status, actualStart, actualEnd, actualDurationMinutes,
    varianceDurationMinutes, lastUpdatedBy, lastUpdatedAt, notes
  participants/{userId}/
    name, role, connectedAt, lastSeenAt
  announcements/{announcementId}/
    message, sentBy, sentAt, displayUntil
```

## Atomic Task List

### 1.4.A — Firebase Realtime Database Setup

- [ ] **Task 1.4.A.1:** Provision Firebase Realtime Database instance via Terraform (`google_firebase_database_instance`)

- [ ] **Task 1.4.A.2:** Set Realtime DB location to `us-central1` (matches Cloud Run region)

- [ ] **Task 1.4.A.3:** Write Realtime DB security rules
  ```json
  {
    "rules": {
      "exercises": {
        "$exerciseId": {
          ".read": "auth != null && auth.token.tenantId != null",
          "tasks": {
            "$taskId": {
              ".write": "auth != null && (auth.token.role == 'admin' || auth.token.role == 'moderator' || root.child('exercises/' + $exerciseId + '/tasks/' + $taskId + '/assignedUserIds/' + auth.uid).exists())"
            }
          },
          "announcements": {
            ".write": "auth != null && (auth.token.role == 'admin' || auth.token.role == 'moderator')"
          },
          "participants": {
            "$userId": {
              ".write": "auth != null && auth.uid == $userId"
            }
          }
        }
      }
    }
  }
  ```

- [ ] **Task 1.4.A.4:** Deploy Realtime DB rules via `firebase deploy --only database`

### 1.4.D — DR Bridge Mode

DR Bridge is a separate route (`/exercises/:id/bridge`) that renders a full-screen live summary — designed for display on a shared screen during the exercise call.

- [ ] **Task 1.4.D.1:** Define DR Bridge data structure
  ```
  Displays (all live-updating from Realtime DB):
  - Exercise name + direction (Failover/Failback)
  - Elapsed time counter (ticking)
  - Current stage name + progress (X/Y tasks)
  - Per-team status summary (team name, tasks: complete/in-progress/not-started/failed)
  - Active announcements (full-width banner)
  - Participant count (connected users)
  - Last updated task (name, status, by whom, how long ago)
  ```

- [ ] **Task 1.4.D.2:** Create `DrBridgeService.buildBridgeSnapshot(exerciseId, tenantId)` — aggregates all data needed for the Bridge view from Firestore (initial load)

- [ ] **Task 1.4.D.3:** Implement `GET /v1/exercises/:id/bridge-snapshot` — returns full initial Bridge data payload

- [ ] **Task 1.4.D.4:** Implement `GET /v1/exercises/:id/bridge` — returns HTML entry point for Bridge view (simple, no auth required for display-only mode with a shareable read token)

- [ ] **Task 1.4.D.5:** Implement Bridge read token: `POST /v1/exercises/:id/bridge-token` — generates a signed, read-only, exercise-scoped token for the Bridge display (no editing, no auth required on the Bridge screen itself)
