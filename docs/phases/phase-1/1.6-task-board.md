# Sub-Phase 1.6 — Exercise Task Board
**Version:** 0.2.0  
**Date:** 2026-03-29  

**Sub-Phase Title:** Exercise Task Board  
**Primary Concern:** Build the live exercise execution interface — the primary operational screen that participants use during a DR exercise. This sub-phase includes the task board, the DR Bridge view, the PWA shell, and all real-time interaction patterns. No admin management screens (those are 1.5). No reports (those are 1.7).  
**Dependencies:** 1.3 (API), 1.4 (Realtime Engine), 1.5 (design system components)  

---

## What This Screen Replaces

The exercise task board replaces the shared Excel workbook that 20+ participants currently edit during a live DR exercise. Every interaction is designed to be faster and less error-prone than the spreadsheet equivalent.

**Core interaction contract:**
- Update a task status in **≤ 2 taps** on mobile
- See another participant's update **within 500ms** without refreshing
- Upload a photo as evidence **without leaving the task row**

### 1.6.C — Task Row Component

This is the most complex component in the application. Every detail matters.

- [ ] **Task 1.6.C.1:** Build `TaskRow` collapsed state
  ```
  Left:   Task ID (monospace font) | Status badge | Task name (truncated)
  Right:  Team chip | Resource chip(s) | Duration indicator
  Locked: Grayed 60% opacity + purple lock icon when predecessors not met
  Optional: Blue dashed left border + "OPTIONAL" chip
  In-Progress: Amber left border (4px) + subtle pulsing animation
  Failed: Red left border + alert icon
  ```

- [ ] **Task 1.6.C.2:** Build `TaskRow` expanded state (tap/click to expand)
  ```
  Full task description (scrollable)
  Start time picker → End time picker (triggers duration display)
  Estimated duration | Actual duration (green if ahead, red if behind) | Variance
  Notes textarea (autosave after 2 seconds of inactivity)
  Evidence section (gallery + upload button)
  Predecessor chips (green = complete, grey = pending, shows Task ID + name)
  ```

- [ ] **Task 1.6.C.3:** Implement status quick-update (≤ 2 taps)
  ```
  Tap status badge → inline status picker appears (segmented control with all 7 statuses)
  Tap new status → optimistic update applied immediately
  API call fires async → if success: nothing (user already sees change)
                       → if fail: revert + Toast "Could not update: [reason]"
  ```

- [ ] **Task 1.6.C.4:** Implement auto-start-time recording: when status changes to `in_progress`, record `actualStart = now()` if not already set

- [ ] **Task 1.6.C.5:** Implement auto-end-time recording: when status changes to `completed` or `failed`, record `actualEnd = now()` if not already set

- [ ] **Task 1.6.C.6:** Implement duration display: `actualDurationMinutes` computed client-side from `actualStart` to `actualEnd`; shown as HH:MM; variance shown with ± prefix and color

- [ ] **Task 1.6.C.7:** Implement notes autosave: debounced 2 seconds; shows "Saving..." → "Saved" indicator; handles offline gracefully (queues save for reconnect)

- [ ] **Task 1.6.C.8:** Implement predecessor lock enforcement in UI: if task is sequential and any predecessor is not `completed`, show lock indicator; status picker is disabled; tooltip lists blocking task IDs

- [ ] **Task 1.6.C.9:** Implement real-time row update animation: when Realtime DB listener fires an update on a task, the row briefly highlights (200ms fade-in highlight)

### 1.6.D — Evidence Upload

- [ ] **Task 1.6.D.1:** Build `EvidenceUploadButton` component — shows camera icon on mobile (triggers `<input type="file" capture="camera">`); file picker on desktop

- [ ] **Task 1.6.D.2:** Implement upload flow:
  ```
  1. User taps camera → file selected
  2. Show upload progress ring on button
  3. Call POST /v1/upload/presign → get Firebase Storage upload URL
  4. Upload directly to Firebase Storage (progress tracked)
  ```
