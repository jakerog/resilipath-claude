# Phase 4 Tasklist — Crisis Communications Manager
**Version:** 0.1.0-DRAFT  
**Date:** 2026-03-29  

---

## EPIC 4.1: Incident Management Core
- [ ] 4.1.1 — `incidents` table: tenant_id, title, description, incident_type (fire/flood/cyber/pandemic/power/other), severity (P1-P4), declared_by, declared_at, status (Active/Monitoring/Resolved/Closed), resolved_at, resolution_summary
- [ ] 4.1.2 — `incident_updates` table: incident_id, author_id, content, update_type (Status/Situation/Action/Resolution), created_at
- [ ] 4.1.3 — Incident declaration UI: quick-declare wizard (< 3 taps/clicks to activate a plan)
- [ ] 4.1.4 — Incident status timeline view: chronological log of all updates, notifications, and acknowledgements
- [ ] 4.1.5 — Incident severity levels with color coding (P1: Critical, P2: High, P3: Medium, P4: Low)
- [ ] 4.1.6 — Multi-incident support: handle multiple simultaneous incidents (e.g., hurricane + power failure)
- [ ] 4.1.7 — Incident handover: reassign incident owner with notification to new owner
- [ ] 4.1.8 — Incident closure workflow: resolution summary → lessons learned → close

## EPIC 4.2: Contact Group Management
- [ ] 4.2.1 — `contact_groups` table: tenant_id, group_name, group_type (Employee/Manager/Executive/Vendor/Regulatory/Government/Customer/Media/Community), description, facility_id (nullable), tags[]
- [ ] 4.2.2 — `contact_group_members` table: group_id, contact_id, is_primary, notification_preferences{email, sms, push, voice}
- [ ] 4.2.3 — `contacts` table: tenant_id, first_name, last_name, title, department, facility_id, email, mobile_phone, work_phone, preferred_channel, notes, is_active, created_from (resource/manual/import)
- [ ] 4.2.4 — Contact group management UI (CRUD, bulk add from Resource table, CSV import)
- [ ] 4.2.5 — Contact deduplication: detect and merge duplicate contacts across groups
- [ ] 4.2.6 — Contact self-service update: tokenized link for contacts to update their own contact information
- [ ] 4.2.7 — Contact group hierarchy: parent/child group relationships (e.g., "Facility A Employees" is child of "All Employees")
- [ ] 4.2.8 — Sync contacts from Module 1 Resource table (bidirectional, with conflict resolution)

## EPIC 4.3: Communication Plan Builder
- [ ] 4.3.1 — `communication_plans` table: tenant_id, plan_name, scenario_type, description, facility_id (nullable), status (Draft/Active/Archived)
- [ ] 4.3.2 — `communication_plan_steps` table: plan_id, step_order, contact_groups[], channels[], template_id, delay_minutes (after previous step or after declaration), require_acknowledgement_before_next (bool), escalation_rules{}
- [ ] 4.3.3 — Communication plan builder UI: visual step-by-step builder (drag-and-drop step ordering)
- [ ] 4.3.4 — Plan test/simulation mode: run a plan in dry-run mode (logs what would be sent, to whom, when — no actual sends)
- [ ] 4.3.5 — Plan version control: maintain history of plan changes with approval workflow
- [ ] 4.3.6 — Plan library: 10 pre-built scenario plans (fire, flood, cyber, pandemic, power, hazmat, active threat, supply chain, data breach, system outage)

## EPIC 4.4: Message Template Engine (Extended from Phase 2)
- [ ] 4.4.1 — `message_templates` table extended: add template_type (Email/SMS/Voice/Push), max_length constraint for SMS (160/320 char), scenario_tag
- [ ] 4.4.2 — Template variable system: `{{incident_title}}`, `{{declared_at}}`, `{{severity}}`, `{{facility_name}}`, `{{next_steps}}`, `{{contact_name}}`, `{{acknowledgement_url}}`
- [ ] 4.4.3 — Multi-channel template set: one "template bundle" links email + SMS + push + voice versions of the same message
- [ ] 4.4.4 — SMS template builder: character counter, multi-part SMS warning, acknowledgement reply code insertion
- [ ] 4.4.5 — Voice script builder: text-to-speech preview, natural language prompt guidelines
- [ ] 4.4.6 — Template approval workflow: templates require approval before becoming active in a plan
- [ ] 4.4.7 — Pre-built template library: 30+ pre-authored templates covering common crisis scenarios

## EPIC 4.5: Multi-Channel Notification Engine
- [ ] 4.5.1 — Email delivery integration (extend Phase 2 engine for mass send with unsubscribe handling)
- [ ] 4.5.2 — SMS delivery integration (Twilio or AWS SNS): send, delivery receipt, opt-out handling
- [ ] 4.5.3 — Push notification delivery (extend Phase 2 PWA push for crisis alerts)
- [ ] 4.5.4 — Voice call delivery (Twilio Programmable Voice): outbound call, play TTS message, press 1 to acknowledge
- [ ] 4.5.5 — Webhook delivery: POST event payload to configured URLs (Slack, Teams, PagerDuty, custom)
- [ ] 4.5.6 — Notification queue: prioritize P1 incidents, guarantee delivery order
- [ ] 4.5.7 — Rate limiting per channel (respect carrier and provider limits)
- [ ] 4.5.8 — Delivery receipt aggregation: collect delivery confirmations across all channels in real-time
- [ ] 4.5.9 — Bulk send performance: 1,000 notifications dispatched within 60 seconds
- [ ] 4.5.10 — Channel failover: if email fails (bounce), automatically retry via SMS for critical contacts

## EPIC 4.6: Acknowledgement Tracking
- [ ] 4.6.1 — `acknowledgements` table: notification_id, contact_id, channel, acknowledged_at, method (link_click/sms_reply/push_tap/voice_keypress), response_text (nullable)
- [ ] 4.6.2 — Unique acknowledgement tokens per notification (tokenized URLs for email links)
- [ ] 4.6.3 — SMS acknowledgement: inbound webhook for SMS replies (STOP/YES/1/2/etc.)
- [ ] 4.6.4 — Voice acknowledgement: DTMF keypress captured in call webhook
- [ ] 4.6.5 — Acknowledgement rate dashboard: real-time % acknowledged per group, per channel, per incident
- [ ] 4.6.6 — Non-responder escalation: configurable threshold (e.g., "if 20% of group hasn't ack'd in 30min, escalate to their manager")
- [ ] 4.6.7 — Escalation notification: auto-send escalation message to defined escalation contacts with non-responder list

## EPIC 4.7: Regulatory Notification Intelligence
- [ ] 4.7.1 — `regulatory_requirements` table: jurisdiction, agency, regulation_name, incident_types[], reporting_deadline_hours, template_id, notes, source_url
- [ ] 4.7.2 — Seed regulatory database: OSHA 300/301 (US), SEC 8-K (public companies), GDPR 72-hour (EU), HIPAA Breach Notification (US healthcare), FDA (food/pharma), EPA EPCRA (hazmat), state-level equivalents for top 10 US states
- [ ] 4.7.3 — Incident-triggered regulatory alerts: on incident declaration, cross-reference incident_type + jurisdiction + industry → surface applicable requirements
- [ ] 4.7.4 — Regulatory checklist UI: per incident, show all triggered requirements with status (Pending/Submitted/Waived), deadline countdown
- [ ] 4.7.5 — Regulatory submission tracking: record submission date, method, recipient, confirmation number
- [ ] 4.7.6 — Deadline alert: automated notifications at T-48h, T-24h, T-4h before regulatory deadline

## EPIC 4.8: Communication Audit Report
- [ ] 4.8.1 — Communication timeline: chronological log of every message sent, channel, recipient, delivery status, acknowledgement
- [ ] 4.8.2 — Summary statistics: total sent, delivered, acknowledged, failed — by group, channel, time window
- [ ] 4.8.3 — Export: PDF audit report (legally formatted, timestamped, digitally signed) and XLSX data export
- [ ] 4.8.4 — Report generation time: < 60 seconds for incidents with up to 10,000 notification events
- [ ] 4.8.5 — Report retention: exported reports stored in tenant audit archive (immutable, 7-year default retention)

## EPIC 4.9: Slack / Microsoft Teams Integration
- [ ] 4.9.1 — Slack app: install in workspace, configure incident notification channel
- [ ] 4.9.2 — Slack notification: post incident declaration + updates as formatted messages to configured channels
- [ ] 4.9.3 — Teams webhook: send adaptive cards for incident notifications to Teams channels
- [ ] 4.9.4 — Slash command (Slack): `/incident declare` to trigger incident declaration from Slack
- [ ] 4.9.5 — Two-way: status updates posted from Slack/Teams are captured in incident log

## EPIC 4.10: Public Status Page
- [ ] 4.10.1 — `status_pages` table: tenant_id, subdomain (status.customerdomain.com), logo, components[] (services/systems being reported on)
- [ ] 4.10.2 — `status_incidents` table: status_page_id, title, status (Investigating/Identified/Monitoring/Resolved), updates[], components_affected[]
- [ ] 4.10.3 — Public status page UI: accessible without login, real-time updates, incident history
- [ ] 4.10.4 — Status page subscribe: customers can subscribe to email/SMS updates for specific components
- [ ] 4.10.5 — Status page embed: embeddable widget for customer-facing websites
