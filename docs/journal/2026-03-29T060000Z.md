## KEY ARCHITECTURAL DECISIONS MADE

### ADR-001: Multi-Tenancy
- **Chosen:** Hybrid model — Schema-per-tenant for Standard/Professional; Database-per-tenant for Enterprise
- **Key driver:** Enterprise security requirements for physical DB isolation; cost efficiency for standard tier
- **Critical implementation note:** ALL DB access must go through `TenantConnectionResolver` — never hardcoded connections

### ADR-002: Real-Time
- **Chosen:** WebSockets for exercise task board; SSE for dashboards and monitoring
- **Key driver:** Exercise board needs bidirectional (user updates go client→server); dashboards are server→client only
- **Message broker:** Redis Pub/Sub for fan-out across server instances
- **Event schema:** Standardized JSON envelope with type, tenant_id, exercise_id, payload, actor, timestamp, sequence

### ADR-003: Authentication
- **Chosen:** Auth0 for authentication; custom RBAC layer in application for authorization
- **Key driver:** Auth0 handles SAML 2.0 natively — critical for Enterprise SSO without custom implementation complexity
- **MFA:** Required for Admin and Moderator roles via Auth0 policy enforcement

## DATA MODEL HIGHLIGHTS

### Critical Design Decisions

**Task duration as generated columns:**
```sql
actual_duration_minutes INT GENERATED ALWAYS AS (
  EXTRACT(EPOCH FROM (actual_end - actual_start))::INT / 60
) STORED
```
This ensures durations are always accurate and never drift from actual timestamps.

**Predecessor enforcement at DB level:**
`task_predecessors` join table with `UNIQUE(task_id, predecessor_task_id)` constraint. Business logic in service layer checks all predecessors are `completed` before allowing status transition to `in_progress` for sequential tasks.

**Evidence file integrity:**
SHA-256 checksum stored at upload time. Files are immutable once uploaded (content-addressed storage key).

**Resource check-in tokenization:**
`confirmation_token UUID UNIQUE` — allows vendors and external resources to confirm availability without a platform login. Token is single-use, expires after exercise start.

## API CONTRACT HIGHLIGHTS

**Total endpoints defined: ~85**

Key design patterns:
- Nested resources for context: `/exercises/:id/stages`, `/stages/:id/tasks`
- Flat resource access: `/tasks/:id` for direct access when exercise context is known
- Specialized PATCH verbs for state transitions: `PATCH /tasks/:id/status`, `PATCH /stages/:id/activate-rollback`
- Tokenized endpoints for no-auth access: `GET/PATCH /checkins/:token`
- WebSocket endpoint: `WS /ws/exercises/:id` with standardized event types

## UX DESIGN DECISIONS

**Primary aesthetic decision: Professional Dark Skeuomorphic**
- Dark mode by default (`#0F172A` background) — justified by early-morning exercise start times observed in Excel data
- Tactile depth/shadow for interactive elements
- Status badges as physical "stamps" with semantic color + icon
- High information density — operators need to see many tasks simultaneously

**Critical interaction decision: < 2 taps for status update**
Optimistic UI update pattern: user sees the change immediately, WebSocket echo confirms it. If echo fails, UI reverts with a visible indicator.

**Go/No-Go decision modal:**
Full-screen takeover — no other UI interaction possible until decision is made. This is intentional to prevent accidental bypassing of critical gates.
