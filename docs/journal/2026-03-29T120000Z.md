## KEY DECISIONS THIS SESSION

### Technology Stack (ADR-004)
Settled on a full TypeScript monorepo:
- **Backend**: Fastify 5 (faster than Express, built-in ajv schema validation, plugin architecture)
- **Frontend**: React 18 + Vite + TanStack Query + Zustand + Radix UI + Tailwind
- **Database**: PostgreSQL 16 with Drizzle ORM (supports dynamic schema for multi-tenancy)
- **Queue**: BullMQ on Redis (already in stack for pub/sub)
- **Email**: Resend with React Email templates (modern API, excellent DX)
- **Monorepo**: pnpm workspaces + Turborepo (incremental builds, remote cache)

**Critical Drizzle insight:** `db.$withSchema(tenantSchema)` is the implementation mechanism for schema-per-tenant. This must be the first architectural pattern established — ALL repositories use this pattern.

### State Machines (ADR-006)
Formally specified all 6 state machines with full transition tables. Key decisions:
- Rollback is NEVER automatic — always a deliberate human decision
- Go/No-Go `no_go` blocks ALL downstream sequential tasks (not just the next one)
- `completed → in_progress` reversal is Admin-ONLY and requires justification — safety valve for live production exercises
- Auto-delay detection runs as a background job every 5 minutes — never changes status, only emits a warning event
- Resource check-in has a 5-escalation-tier system (T-48h, T-24h, T-12h, T-4h, T-1h)

### Testing Philosophy
"Test What Could Kill an Exercise" — prioritization is explicit: state machine unit tests (100% coverage), tenant isolation integration tests (every cross-tenant access attempt), RBAC tests (every role/permission combination), WebSocket real-time tests (200 concurrent connections).

The "Live Exercise Simulation" integration test is a mandatory artifact for every significant feature — simulates 10 concurrent users with different roles progressing through a 20-task exercise end-to-end.

### Security (STRIDE Threat Model)
Major threats identified and mitigated:
- **IDOR (Tenant Isolation):** 404 for cross-tenant resources (not 403 — never confirm existence)
- **Token Forgery:** RS256 (asymmetric) — private key never leaves Auth0
- **Evidence Tampering:** SHA-256 checksum + S3 Object Versioning; Object Lock for Enterprise
- **Phone Number Privacy:** Phone numbers encrypted at application layer (AES-256-GCM) — even DB admins cannot read plaintext
- **Audit Tampering:** DB user for audit table has INSERT only — no UPDATE/DELETE

### Sprint Planning Insights
Phase 0 (40 story points across 3 sprints) delivers the invisible foundation. A new developer should be able to get a working local environment in < 30 minutes — this is an explicit exit criterion.

Phase 1 Sprint 2 is the most technically complex (state machines + WebSocket) — likely a sprint with the most uncertainty. Consider adding 20% buffer or accepting that only the core WebSocket (not all events) is delivered in Sprint 2.

## PATTERNS ESTABLISHED THIS SESSION

### Repository Pattern with Tenant Context
All database access follows this pattern:
```typescript
// CORRECT: Tenant context is explicit
const task = await TaskRepository.findById(db, tenantId, taskId)

// WRONG: Would never be written — no tenant context
const task = await db.query.tasks.findFirst({ where: eq(tasks.id, taskId) })
```

### Error Code System
All business rule violations return structured error codes:
```
TASK_PREDECESSOR_NOT_MET
TASK_NOT_ASSIGNED_TO_USER
INVALID_STATUS_TRANSITION
STAGE_LOCKED
EXERCISE_PHASE_GATE_BLOCKED
GO_NO_GO_APPROVAL_REQUIRED
CROSS_TENANT_ACCESS_DENIED
```
These codes are used in: API responses, unit tests (expect specific code), frontend error handling (map code → user-friendly message), monitoring (alert on unexpected codes).

### "Fail Secure" Pattern
```typescript
// ALL permission checks default to DENY
function checkPermission(role, action, resource): boolean {
  const allowed = PERMISSION_MATRIX[role]?.[action]?.[resource]
  return allowed === true  // undefined → false (not undefined → true)
}
```
